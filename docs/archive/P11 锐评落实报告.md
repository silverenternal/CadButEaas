### ✅ 问题 4: 导出功能未实现（P0 已完成）

**P11 原话**:
> "导出场景：必须保存文件（当前 TODO）"

**落实内容**:

1. **API 客户端导出方法** (`crates/cad-viewer/src/api.rs`)
   ```rust
   pub async fn export_scene(&mut self, path: &str, edges: &[Edge], format: &str) -> Result<(), String> {
       let url = format!("{}/export", self.base_url);
       
       let edge_data = serde_json::to_string(edges)
           .map_err(|e| format!("序列化边数据失败：{}", e))?;
       
       let form = multipart::Form::new()
           .part("edges", multipart::Part::bytes(edge_data.into_bytes()))
           .text("path", path.to_string())
           .text("format", format.to_string());
       
       let response = self.client.post(&url).multipart(form).send().await?;
       let result: ExportResponse = response.json().await?;
       
       if result.success { Ok(()) } else { Err(result.message) }
   }
   ```

2. **本地导出方法**（不依赖后端）
   ```rust
   pub async fn export_scene_local(&self, path: &str, edges: &[Edge], format: &str) -> Result<(), String> {
       let content: Vec<u8> = match format {
           "json" => serde_json::to_string_pretty(edges)?.into_bytes(),
           "bincode" => bincode::serialize(edges)?,
           _ => return Err(format!("不支持的导出格式：{}", format)),
       };
       
       tokio::fs::write(path, content).await?;
       Ok(())
   }
   ```

3. **UI 调用** (`crates/cad-viewer/src/app.rs`)
   ```rust
   pub fn export_scene(&mut self) {
       if let Some(path) = rfd::FileDialog::new().save_file() {
           let path_str = path.display().to_string();
           let api_client = self.api_client.clone();
           let edges = self.edges.clone();
           
           tokio::spawn(async move {
               let mut client = api_client.lock().await;
               client.export_scene(&path_str, &edges, "json").await?;
           });
       }
   }
   ```

## 五、P11 锐评 5.0 总结（2026 年 2 月 26 日）

### 评分：99/100（+1 分 from 98 分）

### 🔪 问题根因分析

**问题现象**:
```
1 检测到文件类型：Dxf
2 处理失败：不支持的文件格式：unknown
```

**核心矛盾**: API 层检测到 `FileType::Dxf`，但 Parser 层说 `unknown`。

**断裂点分析**:

| 层级 | 行为 | 问题 |
|------|------|------|
| API 层 | 通过魔数检测出 `FileType::Dxf` | ✅ 工作正常 |
| 临时文件 | 创建时没有扩展名 `cad_process_5548_job-xxx` | ❌ 问题根源 |
| Parser 层 | `parse_file()` 只认扩展名判断类型 | ❌ 设计缺陷 |

**结论**: 临时文件没有扩展名，Parser 的 `parse_file()` 方法无法识别类型，返回 `UnsupportedFormat { format: "unknown" }`。

---

### 🛠️ 落地方案（采用方案一）

**方案一：临时文件保留扩展名**（⭐ 推荐，改动最小）

**修改点**: `crates/orchestrator/src/api.rs` 第 257-265 行

**思路**: 创建临时文件时，根据检测到的 `detected_type` 添加对应扩展名：
```
cad_process_5548_job-xxx  →  cad_process_5548_job-xxx.dxf
```

**实现代码**:

1. **添加辅助函数** (api.rs:441-448)
   ```rust
   /// 将 FileType 转换为文件扩展名
   fn file_type_extension(file_type: &FileType) -> &'static str {
       match file_type {
           FileType::Dxf => "dxf",
           FileType::Pdf => "pdf",
           FileType::Unknown => "unknown",
       }
   }
   ```

2. **修改临时文件创建逻辑** (api.rs:259-263)
   ```rust
   // 创建临时文件（保留扩展名以便 Parser 识别）
   let temp_dir = std::env::temp_dir();
   let temp_file_name = format!("cad_process_{}_{}.{}",
       std::process::id(),
       uuid_simple(),
       file_type_extension(&detected_type)
   );
   let temp_path: PathBuf = temp_dir.join(&temp_file_name);
   ```

**优点**:
- ✅ 改动最小（只需改 5 行代码）
- ✅ 不破坏现有架构
- ✅ Parser 层无需修改

**缺点**:
- ⚠️ 临时文件名略长

## 六、P11 锐评 6.0 总结（2026 年 2 月 26 日·前端渲染优化）

### 🔪 锐评：前端渲染的 7 大致命伤

**问题 1**: 坐标变换系统存在精度隐患
- **症状**: f64→f32 强制转换丢失精度，大坐标文件（如 UTM 坐标）渲染抖动
- **修复**: 采用"先减后乘"策略，减少大数相加导致的精度损失
- **代码**: `canvas.rs` 第 19-32 行 `world_to_screen()` 方法

**问题 2**: 网格绘制性能灾难
- **症状**: 每次 update() 重绘整个网格，zoom=0.1 时网格线过密
- **修复**: 限制网格密度 `.clamp(20.0, 500.0)` + 视口裁剪
- **代码**: `canvas.rs` 第 381-413 行

**问题 3**: 线段选择算法效率低下
- **症状**: O(n) 暴力遍历，10000 条边时点击延迟 100ms+
- **修复**: 视口内选择 + 提前退出（距离<1 像素直接返回）
- **代码**: `canvas.rs` 第 292-336 行

**问题 4**: 渲染状态管理混乱
- **症状**: 15 个状态字段挤在 CadApp struct，违反单一职责
- **计划**: P2 阶段重构为状态机（需后端 API 配合）

**问题 5**: 颜色系统过于简陋
- **症状**: 硬编码蓝色，无图层颜色映射/线宽/线型支持
- **计划**: P2 阶段实现图层配置（需后端 API 配合）

**问题 6**: 没有视口优化
- **症状**: 每次绘制所有边，不管是否在视口内
- **修复**: 添加 `line_in_viewport()` 快速拒绝测试
- **代码**: `canvas.rs` 第 47-62 行 `line_in_viewport()` 方法

**问题 7**: 错误处理缺失
- **症状**: 只检查 Completed 状态，Partial 状态被当错误
- **修复**: 添加超时控制（30 秒）+ Partial 状态处理
- **代码**: `api.rs` 第 98-143 行 `load_file()` 方法

```rust
/// 变换实体（应用缩放、旋转、平移）
fn transform_entity(
    entity: &RawEntity,
    scale: [f64; 3],
    rotation_deg: f64,
    translation: Point2,
) -> RawEntity {
    let rotation_rad = rotation_deg.to_radians();
    let cos_rot = rotation_rad.cos();
    let sin_rot = rotation_rad.sin();

    fn transform_point(point: Point2, scale: [f64; 3], cos_rot: f64, sin_rot: f64, translation: Point2) -> Point2 {
        // 1. 缩放
        let scaled = [point[0] * scale[0], point[1] * scale[1]];
        // 2. 旋转（绕原点）
        let rotated = [
            scaled[0] * cos_rot - scaled[1] * sin_rot,
            scaled[0] * sin_rot + scaled[1] * cos_rot,
        ];
        // 3. 平移
        [rotated[0] + translation[0], rotated[1] + translation[1]]
    }

    match entity {
        RawEntity::Line { start, end, .. } => {
            RawEntity::Line {
                start: transform_point(*start, scale, cos_rot, sin_rot, translation),
                end: transform_point(*end, scale, cos_rot, sin_rot, translation),
                ..entity.clone()
            }
        }
        // ... 其他类型
    }
}

/// 计算场景边界
fn calculate_scene_bounds(&self) -> Option<([f64; 2], [f64; 2])> {
    if self.edges.is_empty() {
        return None;
    }

    let mut min_x = f64::INFINITY;
    let mut max_x = f64::NEG_INFINITY;
    let mut min_y = f64::INFINITY;
    let mut max_y = f64::NEG_INFINITY;

    for edge in &self.edges {
        min_x = min_x.min(edge.start[0]).min(edge.end[0]);
        max_x = max_x.max(edge.start[0]).max(edge.end[0]);
        min_y = min_y.min(edge.start[1]).min(edge.end[1]);
        max_y = max_y.max(edge.start[1]).max(edge.end[1]);
    }

    // 添加 10% 边距
    let margin_x = (max_x - min_x) * 0.1;
    let margin_y = (max_y - min_y) * 0.1;

    Some((
        [min_x - margin_x, min_y - margin_y],
        [max_x + margin_x, max_y + margin_y],
    ))
}

/// 自动适配场景边界
pub fn fit_to_scene(&mut self) {
    if let Some((min, max)) = self.calculate_scene_bounds() {
        let scene_width = max[0] - min[0];
        let scene_height = max[1] - min[1];

        // 获取画布尺寸（假设默认 800x600）
        let view_width = 800.0;
        let view_height = 600.0;

        // 计算适配缩放
        let zoom_x = view_width / scene_width;
        let zoom_y = view_height / scene_height;
        let zoom = zoom_x.min(zoom_y).clamp(0.1, 10.0);

        self.zoom = zoom as f32;

        // 计算场景中心
        let scene_center_x = (min[0] + max[0]) / 2.0;
        let scene_center_y = (min[1] + max[1]) / 2.0;

        // 设置平移使场景居中
        self.pan = egui::Vec2::new(
            -scene_center_x as f32 * self.zoom,
            -scene_center_y as f32 * self.zoom,
        );

        self.add_log(&format!(
            "已适配视图：缩放 {:.0}%, 场景范围 [{:.1}, {:.1}] × [{:.1}, {:.1}]",
            self.zoom * 100.0,
            min[0], max[0],
            min[1], max[1]
        ));
    }
}

           let parse_request = parser::service::ParseRequest::new(&input.path);
           let parse_result = self.parser.process(parse_request).await?;
           
           // 2. 提取多段线
           let entities = parse_result.into_entities();
           let polylines = Self::extract_polylines_from_entities(&entities);
           
           // 3. 构建拓扑：通过 TopoService::process() 调用
           let topo_request = topo::service::TopoRequest {
               geometry_json: serde_json::to_string(&polylines)?,
           };
           let topology_result = self.topo.process(topo_request).await?;
           
           // 4. 验证：通过 ValidatorService::process() 调用
           let validate_request = validator::service::ValidateRequest::new(
               serde_json::to_string(&scene)?
           );
           let validation_report = self.validator.process(validate_request).await?;
           
           // 5. 导出：通过 ExportService::process() 调用
           let export_request = export::service::ExportRequest::new(scene.clone());
           let export_result = self.export.process(export_request).await?;
           
           Ok(ProcessResult {
               scene,
               validation: validation_report,
               output_bytes: export_result.bytes,
           })
       }
   }
   ```

**交付物**:
- `parser/src/service.rs` - ParseRequest/ParseResult 类型，process() 返回完整结果
- `topo/src/service.rs` - TopoRequest/TopologyResult 类型，process() 返回完整结果

### 🔧 架构哲学

**P11 锐评原文**:
> "架构是服务于业务的，不是束缚业务的。
> 如果你发现 EaaS 让你的代码更清晰，就坚持它。
> 如果你发现 EaaS 让你的代码更复杂，就简化它。
> 最可怕的是：为了架构而架构，为了模式而模式。"

**我们的选择**:
1. **坚持 EaaS 架构** - 因为它带来了：
   - 统一的服务契约
   - 统一的健康检查
   - 统一的指标收集
   - 服务可组合性
   - 服务可治理性

2. **务实的设计** - 平衡架构优雅性和性能：
   - `process()`: 通过统一接口编排（EaaS 演示）
   - `process_file()`: 直接调用业务方法（高性能）
   - 两者并存，按需选择

3. **微服务就绪** - 未来可以：
   - 将任何服务独立部署为 HTTP/gRPC 服务
   - 透明切换本地/远程调用
   - 服务熔断降级
   - 服务指标监控
