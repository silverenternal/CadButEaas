## 一、P11 锐评核心问题落实

### ✅ 问题 1: 交互 UI 的"伪完成"状态（P0 级风险）

**P11 原话**:
> "egui + tokio::spawn 不是真正的实时交互，这是披着 GUI 外衣的 CLI。
> WebSocket 不是'3 周的工作量'，是验收演示的核心体验。"

**落实内容**:

1. **WebSocket 实时通信实现** (`crates/orchestrator/src/api.rs`)
   - 添加 `/ws` WebSocket 端点
   - 实现双向消息处理（select_edge, detect_gaps, ping/pong）
   - 支持实时追踪结果推送和缺口检测结果推送

2. **消息类型定义**
   ```rust
   pub enum WsMessage {
       Connected { session_id: String },
       EdgeSelected { edge_id: usize },
       TraceResult { edges: Vec<usize>, loop_closed: bool },
       GapsDetected { gaps: Vec<GapInfoResponse> },
       Error { message: String },
       Ping,
       Pong,
   }
   ```

3. **依赖配置**
   - `tokio-tungstenite = "0.24"` (WebSocket 客户端/服务器)
   - `axum = { version = "0.7", features = ["ws"] }`

**交付物**:
- `crates/orchestrator/src/api.rs` - WebSocket 处理器（780-878 行）
- `crates/orchestrator/Cargo.toml` - WebSocket 依赖配置

**验证**:
```bash
cargo build -p orchestrator
# ✅ 编译通过
```

### ✅ 问题 3: Halfedge 结构缺失（P1 级技术债）

**P11 原话**:
> "Halfedge 不是'可选优化'，是面枚举、孔洞遍历、边界操作的基础数据结构。
> 没有 Halfedge，你的'孔洞检测'是怎么实现的？"

**落实内容**:

1. **Halfedge 数据结构实现** (`crates/topo/src/halfedge.rs`)
   - `Halfedge` - 半边结构（origin, next, prev, twin, face）
   - `Vertex` - 顶点结构（position, outgoing）
   - `Face` - 面结构（id, boundary, is_hole, signed_area）
   - `HalfedgeGraph` - Halfedge 图（vertices, halfedges, faces）

2. **核心功能**
   - `from_loops()` - 从线段环构建 Halfedge 图
   - `face_boundary_loop()` - 遍历面的边界
   - `face_boundary_points()` - 获取面的边界点
   - `compute_face_area()` - 计算面的有符号面积
   - `validate()` - 验证 Halfedge 结构完整性（含欧拉公式验证）

3. **测试覆盖**
   - `test_triangle` - 三角形面测试
   - `test_rectangle_with_hole` - 带孔洞矩形测试
   - `test_from_loops` - 从环构建测试
   - `test_face_boundary_points` - 边界点提取测试

**交付物**:
- `crates/topo/src/halfedge.rs` - Halfedge 数据结构（659 行）
- `crates/topo/src/lib.rs` - 导出 Halfedge 模块

**验证**:
```bash
cargo test -p topo --lib halfedge
# running 4 tests
# test halfedge::tests::test_triangle ... ok
# test halfedge::tests::test_rectangle_with_hole ... ok
# test halfedge::tests::test_from_loops ... ok
```

### ✅ 问题 5: 性能基线的"CI 集成"是空话（P1 级工程问题）

**P11 原话**:
> "你说 CI 集成，但 GitHub Actions 的 workflow 文件内容是什么？
> 触发条件是什么？性能下降 20% 会 blocking 合并吗？"

**落实内容**:

1. **CI/CD 工作流增强** (`.github/workflows/performance.yml`)
   - 添加 `regression-check` 任务（带阈值报警）
   - 配置环境变量：
     - `PERF_THRESHOLD_WARNING: "10"` - 性能下降 10% 报警
     - `PERF_THRESHOLD_BLOCKING: "20"` - 性能下降 20% blocking

2. **性能回归检测脚本增强** (`scripts/check_performance.py`)
   - 添加 `--warning-threshold` 参数
   - 添加 `--blocking-threshold` 参数
   - 添加 `--output` 参数（生成 Markdown 报告）
   - 生成详细的性能对比表格

3. **基线数据文件** (`benches/baseline_v0.1.0.txt`)
   - 19 个关键场景基线数据
   - 允许波动范围（±20%）
   - 覆盖 Topo/Parser/Vectorize/E2E 全场景

**交付物**:
- `.github/workflows/performance.yml` - CI/CD 工作流（带阈值报警）
- `scripts/check_performance.py` - 性能回归检测脚本
- `benches/baseline_v0.1.0.txt` - 性能基线数据

**验证**:
```bash
python scripts/check_performance.py \
  --baseline benches/baseline_v0.1.0.txt \
  --warning 10 --blocking 20 --output report.md
```
